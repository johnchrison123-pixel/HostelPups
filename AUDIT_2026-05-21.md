## A. Promised features in CLAUDE.md not yet implemented

CLAUDE.md says Phase 0 = "Foundation … placeholder auth/listing pages" and Phase 1 = "Supabase schema, real auth (phone OTP), owner listing CRUD, real search + filters, listing detail with photos". The Instructions History also captures specific owner asks:

- 🔴 **Phone OTP not wired.** `SignupForm`, `OwnerSignupForm`, `LoginForm` all use `setTimeout(700)` stubs. No call to `sb.auth.signInWithOtp` / `verifyOtp` anywhere. (refs: `src/components/auth/SignupForm.tsx:110`, `:125`; `OwnerSignupForm.tsx:130`, `:147`; `LoginForm.tsx`)
- 🔴 **No real Supabase queries anywhere.** Only `getSupabase()` is exported; nothing in `app/**` imports `sb`. Featured listings, search results, listing detail, and the new owner pages all read from `src/lib/mockListings.ts`.
- 🔴 **Migrations not applied to the actual Supabase project.** `0001_initial_schema.sql` → `0004_calls_schema.sql` exist locally; founder must run them via Supabase CLI. (Out of scope per task spec, but flagged so it's not forgotten.)
- 🟠 **Owner CRUD writes are stubbed.** Task 2 scaffolding (this round) intentionally has no Supabase writes — the listing form, profile form, and pause/delete buttons all just `console.log` + show a toast / fake success. This is by design for the current phase but must be wired before launch.
- 🟠 **"In-app call" feature from May 21 instruction.** Migration shipped this round (`0004_calls_schema.sql`); WebRTC client code is still entirely absent. Banner on `/owner/calls` says "Phase 2".
- 🟠 **"Keep saved login data."** No `persistSession` consumer in the auth flows (Supabase client has it set, but since auth isn't wired, this is moot until OTP lands).
- 🟠 **"Photos/profile/message/call data."** Listing photo upload is UI-only — Supabase Storage `listing-photos` bucket exists per `0003_storage_setup.sql` but no upload call. Renter avatar upload is missing entirely.
- 🟠 **"Filter in left vacant section."** Search page has FilterSidebar with city / area / type / gender / tag — but **no price (budget) filter, no amenities filter, no vacancy-availability filter**, despite the search-page meta description advertising "search by … budget … and amenities" (`src/app/search/page.tsx:29`).
- 🟡 **Block obfuscated contact tricks.** Already implemented in `redactContactInfo()` (`src/lib/utils.ts:236`) with 28 test cases — but it's never called from a message-send path because messaging isn't wired. So the function works in isolation but isn't load-bearing yet.
- 🟡 **Owner "manage profiles, cover page, images, descriptions, vacancies."** Manage UI scaffolding shipped this round under `/owner/*`, but cover-image management, photo reorder persistence, and vacancy editing are all PENDING writes.

## B. Dead / broken stuff

### 🔴 Broken anchors in Footer
- `Footer.tsx` lines 55-56 link to `/for-owners#full-service` and `/for-owners#self-serve`, but `for-owners/page.tsx` only defines `#pricing`, plan-card-level `#${p.id}` (which would need to be `id="full-service"` etc.), and `#verification`. The two named anchors don't exist. Either rename the section ids on `for-owners` or fix the Footer hrefs to `#pricing`.

### 🔴 No `not-found.tsx`
- `src/app/notFound()` is called from `pg/[city]/[slug]/page.tsx:147` but there is no `app/not-found.tsx`. Next.js falls back to its default white-page 404 — generic, unbranded, no nav back. This is bad UX for typo-redirected users and bad SEO (no `noindex` markup, no internal links to recover).

### 🔴 Cities in `CITY_NAMES` without landing pages
- `CITY_NAMES` (in `src/lib/site.ts:54`) includes `hyderabad`, `pune`, `mumbai`, `delhi`, `noida`, `gurgaon`, `kollam`, `kannur`, `kottayam`, `palakkad` — and the listing form / signup form expose all of them as city options. But there is **no** `/pg-in-hyderabad`, `/pg-in-pune`, etc. If an owner registers in Pune (allowed by the dropdown) and the system later tries to link them to a city page, the link 404s. **Either prune `CITY_NAMES` or add stub city pages with "coming soon" framing.**
- Note: `/cities` page explicitly handles this for the 4 Kerala "coming soon" cities, but the rest aren't even on the list.

### 🟠 Forms submit to stubs without server feedback
- `SignupForm`, `OwnerSignupForm`, `LoginForm` all `setTimeout` then "succeed" with no real OTP being sent. Users will think they're signed up and bounce off the success screen — and there'll be no Supabase row to back it up. **At minimum, the success screens should say "Demo — auth not wired" until Phase 1A lands**, otherwise this is misleading.
- New `ListingForm.tsx` already says PENDING clearly in the success state; older auth forms do not.

### 🟠 Owner CRUD buttons that "do nothing"
- `/owner/listings`: Pause + Delete table buttons are bare `<button>` with no handler. `title="Pause listing (PENDING)"` helps a sighted developer but the user just sees no feedback.
- `/owner/listings`: Status filter pills (All / Live / Paused / Pending / Draft) are visually styled as active/inactive but have no `onClick` or `Link` — clicking them is a no-op.
- `/owner/profile`: "Upload logo" and "Save changes" buttons have no real handler.
- These are intentional placeholders, but a single "PENDING Phase 1B — wiring next round" banner at the top of each page (instead of buried captions) would help.

### 🟠 Stub social URLs in metadata
- `SITE.social.{instagram,facebook,twitter,youtube,linkedin}` (in `site.ts:25-31`) all point to `https://instagram.com/hostelpups` etc. — but those handles aren't owned. They're surfaced in Footer + JSON-LD `Organization.sameAs`, which is technically lying to Google about identity. **Either claim the handles or comment them out from `sameAs` until claimed.**

### 🟡 `SITE.phone` still placeholder
- `+91-XXXXX-XXXXX` in `src/lib/site.ts:13`. Not currently rendered anywhere — no impact today, but it'll show as soon as a Contact page renders it.

### 🟡 Search page advertises "budget" filter that doesn't exist
- Meta description in `src/app/search/page.tsx:29` says "filter by … budget …", but FilterSidebar has no price slider/input. Either add it or trim the description.

### 🟡 "Open chat" goes to a placeholder messages screen
- `/owner/inquiries`'s "Open chat" buttons link to `/messages`, which is currently the "Sign in to see your conversations" screen — owners signed in won't see anything. Same goes for renter side. Whole messaging UI is PENDING.

### 🟢 `/admin/*` referenced in robots but no admin page
- `robots.ts` disallows `/admin/` but no admin pages exist yet. Harmless precaution, but worth tracking.

## C. UX gaps (things users will hate)

### 🔴 No global loading or error boundaries
- No `app/loading.tsx` (skeleton/spinner for slow segments).
- No `app/error.tsx` (graceful error UI). A throw in any page = the default white screen.

### 🔴 Search results have no skeleton state
- Server-render is fast today because data is in-memory, but as soon as Supabase queries land in Phase 1B, the search page will appear "stuck" with nothing for ~1-2s.

### 🟠 Delete listing flow needs a proper confirmation modal
- `ListingForm.tsx` uses `window.confirm()` — works but ugly and inconsistent with the design system. Replace with a Card-based dialog (see for-owners/contact patterns for design language).

### 🟠 No toast / notification primitive yet
- The success screens after OTP and after listing submit have inline cards; there's no top-level toast container to confirm shorter actions (e.g. "Listing paused", "Saved to favourites", "Logout successful"). Build a small portal-mounted toast in `src/components/ui/Toast.tsx` before Phase 1B.

### 🟠 No 404 handling for listing detail
- `pg/[city]/[slug]/page.tsx` calls `notFound()` if no listing matches — but there's no `not-found.tsx` (covered above). High-frequency case once Supabase data lands and slugs change.

### 🟠 Owner logout flow doesn't actually log out
- `OwnerSidebar.tsx` "Sign out" links to `/owner/login` but doesn't call `sb.auth.signOut()`. (Acceptable in scaffolding phase, but worth noting.)

### 🟠 Mobile owner navigation not surfaced from header
- `OwnerSidebar.tsx` does provide a mobile drawer with hamburger — good. But on `/owner/login` and `/owner/signup`, there's no sidebar at all (they use `AuthSidePanel` instead), so a newly registered owner with no listings has no nav until they hit a dashboard route. Probably fine.

### 🟡 No "are you sure?" before navigating away from the listing form
- A user 4 steps in who accidentally clicks Header → "Find PG" loses everything. Add a `beforeunload` listener inside `ListingForm.tsx` when `data` has been modified.

### 🟡 Inquiries page status filter missing
- `/owner/inquiries` shows mixed open/replied/closed inquiries without any filter chips at the top.

### 🟡 No empty state when zero inquiries
- Listings page has an empty state (well — empty branch is in the code; with mock data we never see it). Inquiries page has no empty state at all.

### 🟡 No timezone awareness on `timeAgo()`
- `src/lib/utils.ts:444` uses `Date.now()` minus the string-parsed date, which is timezone-sane, but the seed dates in mock data are UTC. Will look fine until owners in IST sign up and we display "in 5 hours" because of clock skew.

### 🟢 Color contrast on `--color-ink-subtle`
- Hasn't been measured. Some captions (esp. "Pending Phase 2 data wiring") may be < AA against off-white background. Quick `npx pa11y-ci` pass before launch.

## D. SEO gaps

### 🟠 `/cities` not in sitemap
- `cities` IS in `sitemap.ts:33` — actually it is. Good. (Self-corrected.)

### 🟠 City pages missing for cities in `CITY_NAMES`
- Recap from B: Hyderabad / Pune / Mumbai / Delhi / Noida / Gurgaon / Kollam / Kannur / Kottayam / Palakkad have no landing pages. These are SEO golden eggs — "PG in Bangalore" is a #1 search-volume query and there's an empty page slot for it... oh wait, `/pg-in-bangalore` exists. But "PG in Pune", "PG in Hyderabad", "PG in Mumbai" are open turf and we have zero pages there.
- **Recommendation:** add stub `pg-in-{city}` pages for the 6 Tier-1 cities with "Launching soon" copy + "join waitlist" CTA. Even thin pages rank for low-competition long-tail queries.

### 🟠 Listing detail page uses gradient placeholders only — no `next/image`
- `pg/[city]/[slug]/page.tsx:207-244` renders a CSS gradient div instead of `<Image>`. Once real photos land in Phase 1B, this needs `next/image` for AVIF/WebP / responsive sizes / proper alt. Already noted in code comments but worth surfacing.

### 🟠 No `priceFrom` populated in JSON-LD when min price is 0
- `lodgingSchema()` in `seo.ts:166` outputs `priceRange: "₹{n}+ per month"` only if `priceFrom > 0`; when 0 it falls back to "Price on request". Some listings have rooms in db where minimum-price logic might give 0. Low risk today (mock data always has prices), but worth a test.

### 🟡 OG image `og-default.png` may not exist
- Referenced from `SITE.ogImage = "/og-default.png"` (`site.ts:22`) — couldn't verify in `public/` (didn't check). If missing, every Open Graph share shows a broken image. **Founder action: verify `public/og-default.png` exists, ≥1200x630, < 200KB.**

### 🟡 Owner pages excluded from sitemap (correct) but not consistently from robots
- `robots.ts` (this round patched) now lists all owner routes. Verify by hitting `/robots.txt` after deploy.

### 🟢 Canonical URLs all correct
- Every page goes through `buildMetadata()` which sets `alternates.canonical`. No issues found.

## F. Mobile responsiveness specific issues

### 🟠 Owner listings table is horizontally cramped on phone
- `/owner/listings`: the table hides Status / Vacancies / Updated columns at `sm:`, `md:`, `lg:` breakpoints respectively, leaving just Listing + Actions on phone. Works, but on a 360px screen the action icon row can wrap awkwardly. Consider switching to a card-stack layout on `<sm` instead of hiding columns.

### 🟠 Listing form room rows
- `ListingForm.tsx` rooms step uses `grid sm:grid-cols-[2fr_1fr_1fr_1fr_auto]` — on phones (no `sm:`) it stacks vertically, which is correct, but the AC checkbox + delete button row at the bottom is fragile. Test on a real 360px viewport.

### 🟠 Hero section text size
- `src/components/marketing/Hero.tsx` — h1 sizes weren't measured. Founder said "8hr vibecoding sessions" so let's not nitpick, but check that the h1 doesn't go below 28px on phones (Lighthouse complains under 28).

### 🟡 Mobile bottom nav covers content
- `<main>` has `pb-16 lg:pb-0` (in `layout.tsx:54`) which accounts for the bottom nav. Owner pages also use `OwnerLayout` which scrolls correctly. Tested.

### 🟡 Touch target sizes
- Search-page filter pills use `min-h-[28px]` (`src/app/search/page.tsx:289`) — below the WCAG 2.2 minimum of 44x44 for touch targets. They're text links so partially exempt, but for the X-to-remove icons inside, 28px is too tight. Bump to 36px+.
